    About

    How the Model Works

    1. URL or metrics entered → 2. Public metrics checked (URL mode) → 3. Form filled → 4. API receives JSON → 5. ML model predicts → 6. Risk score returned

    The dashboard sends metrics to POST /predict. The trained Random Forest returns outage_predicted and outage_probability, then explanation cards highlight risky inputs.

    How to Predict Any Website

    1. Type a public website URL and click Check Website Metrics.
    2. The app measures status code, response time, latency, and simple error rate from 5 probes.
    3. Those values auto-fill the prediction form.
    4. CPU and memory are private — use real values if you own the website, or edit the demo defaults.
    5. Click Predict Outage Risk.
    6. Read outage probability and the risk explanation cards.

    What Public URL Checking Can and Cannot Do

    Can measure:
    • HTTP status code
    • Response time
    • Basic latency (P95 from probes)
    • Whether the website responds
    • Simple error rate from repeated checks
    Cannot measure unless you own the website:
    • Real CPU usage
    • Real memory usage
    • Internal request count
    • Server logs
    • Full application error rate
    • Infrastructure health

    Real-World Workflow

    1. Azure Monitor or Application Insights collects live website metrics.
    2. Metrics are sent to the prediction API.
    3. The model predicts outage probability.
    4. If risk is high, the DevOps team gets alerted.
    5. Engineers investigate before the website goes down.

    Frequently Asked Questions

    What is Website Outage Prediction?
    A machine learning service that estimates the risk of a website outage from monitoring metrics such as response time, error rate, CPU, and memory.
    How was this application built?
    Our team built a full MLOps pipeline on Azure: Python for data and training, FastAPI for the API, a Random Forest model, Docker and GitHub Actions for deployment, and monitoring with Evidently and OpenRouter.
    What does this dashboard do?
    It is a single control center to run the local pipeline, check public website URLs, predict outage risk, and view system health — without running each script manually.
    How does URL checking work?
    The app sends five public HTTP probes to your URL, measures status and latency, auto-fills the prediction form, and saves successful checks in a server-side history list.
    What is the end-to-end MLOps flow?
    Data is versioned and ingested, the model is trained and evaluated behind a quality gate, registered in Azure ML, deployed via Docker to ACR/ACI/AKS, then monitored for drift and explained with AI reports.
    Which API endpoints should I know?
    GET /health checks readiness, POST /predict returns outage risk, POST /check-url-metrics probes a URL, and /docs is the interactive API reference.
    Can the model predict from a URL alone?
    No. It needs numeric metrics. URL check supplies public signals only; you still send those values to /predict.
    Where do CPU and memory values come from?
    Public URL checks cannot read real server CPU or memory. The dashboard uses demo defaults (50%) unless you enter values from Azure Monitor or your own infrastructure.
